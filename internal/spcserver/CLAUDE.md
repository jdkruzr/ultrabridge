# internal/spcserver

Last verified: 2026-05-25

Device-facing reimplementation of the Supernote Private Cloud (SPC) protocol so an unmodified Supernote device talks to UltraBridge as if it were the real SPC server.

**Status:** Phase 1 complete and validated on real hardware (Supernote Nomad SN078C10034074, 2026-05-23) — auth/login, the Engine.IO socket, and bidirectional task sync (incl. instant web→device push) all work. Phase 2 (file listing + capacity, **read path**) is complete and hardware-validated (2026-05-23): the device browsed its native six-folder tree, walked it via `query_v3`, showed a sane capacity meter, and saw no Boox content. Phase 3 (file **download**) is complete and hardware-validated (2026-05-23): `download_v3`/`generate/download/url` mint presigned URLs and `GET /api/oss/download` (tokenless — signature-auth'd) streams the bytes (Range-aware), round-trip MD5 byte-exact on device. This closes the Phase 2 `download_v3`→404 boundary. **Gotcha proven on hardware:** the device sends `download_v3` id as a quoted string (typed `*int64` it silently 404s every download — see §8). Phase 4 (file **upload** + **mutations**, write path) is complete and hardware-validated (2026-05-25): the device uploads a `.note` (lands byte-exact, OCR'd, searchable), renames/moves (`move_v3`, id-stable), copies (`copy_v3`, fresh id), and deletes (`delete_folder_v3` → soft-delete to `.recycle/`). **Phase 4 is purely additive** — no legacy SPC code removed (coexistence; see memory `project_spc_coexistence_no_premature_cutover`). **Two hardware-only wire gotchas** (decompiled annotations were wrong; see §8): upload `path` is the full path in `apply` but parent+`fileName` in `finish`; `move_v3`/`copy_v3` `to_path` is the **full destination path**, not a parent dir — both bit us as `…/X.note/X.note` double-nesting. **UB-as-SPC uses its OWN dedicated `UB_SPC_FILE_ROOT`** (e.g. `/mnt/supernote/ub_sn_files`), NOT the real SPC's data dir (memory `project_spc_dedicated_file_tree`). Recycle CRUD/search are Phase 5. **Phase D (digests / "summary") is complete and hardware-validated (2026-05-25):** the full `F_SummaryController` surface (item/group/tag CRUD + `query/summary/{,/hash,/id,/group}` + `.mark` handwriting blobs over the OSS path) is implemented over `internal/digeststore` (canonical store) via `handlers/summary.go`. Validated on the Nomad both directions: push items/groups, `.mark` upload byte-exact, pull (device fetched server-injected digests incl. `createTime`/`updateTime` millis + `download/summary` → `.mark` render), and device delete/update (incl. update-adds-handwriting + move-between-groups). Wire findings in `docs/spc-protocol.md §8` (item identity = `metadata.unique_identifier`; author in metadata; digest delete is device-authoritative — server-only deletes don't propagate, need a D2 tombstone). Purely additive — when no `DigestStore` is wired the three query endpoints fall back to the old empty-success stubs and the write endpoints 404 (pre-Phase-D behavior). Gated by `UB_SPC_MODE` (default `client` = no listener; `server` = bind `:8089`).

## Layout

- `server.go` — HTTP + Engine.IO server on one listener; `registerRoutes` wires everything. `settingstore.go` — notedb-backed `auth.SettingStore`.
- `envelope/` — leaf package: `BaseVO` + `WriteJSON`/`WriteError`. **Lives here, not in package spcserver, to break the import cycle** (handlers/auth/server all import it; it imports nothing internal).
- `auth/` — HS256 JWT mint/verify + `x-access-token` middleware (+ userId harvest).
- `login/` — randomCode store, password recipe (`sha256Hex(md5Hex(raw)+code)`), `ResolveUserID`.
- `dto/` — request DTOs / response VOs, field names verbatim from decompiled source.
- `mapping/` — `taskstore.Task ↔ dto.SPCTask` at the controller boundary (no second store).
- `socketio/` — Engine.IO v3 codec, connection registry, websocket handler.
- `groups/` — `GroupProvider` seam; single synthesized group today (multi-collection deferred).
- `dedup/` — ResubmitCheck (1s TTL). `notify/` — STARTSYNC notifier over the socket registry.
- `fileids/` — leaf package owning the `spc_file_ids` table (path↔id Long registry + lazy MD5 cache); migrated package-locally in server mode (à la `mcpauth.Migrate`), **not** by `notedb.Open`.
- `capacity/` — leaf package: du-style usage Meter with a 60s cache. **Its own package (not `capacity.go` at the root) to avoid the handlers↔spcserver import cycle** — same reason as `envelope/`.
- `oss/` — leaf package: presigned-URL signing primitive (`EncryptPath`/`DecryptPath` = base64url-no-pad; `Signer` = plain SHA-256 with the secret concatenated, **not** HMAC, per `SignVerifier.java`; 24h download / 30min upload windows). Golden-master-tested against §6's real-SPC vectors. Used by download (Phase 3) and upload (Phase 4).
- `staging/` — leaf-ish package (Phase 4): the `spc_uploads` table (migrated package-locally in server mode) + `Store` (stage bytes to `<FileRoot>/.staging/<innerName>`, verify md5/size, atomic-promote to the target, orphan-sweep). The promotion target comes from upload/**finish** (`path`+`fileName`), not the apply row.
- `handlers/` — equipment, login/challenge/boot, schedule (group/task/sort), `files.go` (Phase 2 read path: sync-session, list_folder/_v3, query_v3/by-path_v3, capacity/query, get_space_usage, real create_folder_v2 + deleteApi stub), `download.go` (Phase 3: download_v3 + generate/download/url mint presigned URLs; DownloadStream serves bytes with Range support), `upload.go` (Phase 4: upload/apply mints an innerName + presigned URL, UploadStream sinks multipart bytes to staging, Finish verifies+promotes+kicks OCR), `mutation.go` (Phase 4: delete_folder_v3 soft-delete to `.recycle/`, move_v3, copy_v3), and `summary.go` (Phase D: the digest surface — item/group/tag CRUD + queries + `.mark` apply/download; thin DTO↔`digeststore.Digest` adapter behind the `DigestStore` interface). `schedule.go` still carries `SummaryStub` for the no-DigestStore fallback.

Storage for digests lives **outside** this package in `internal/digeststore` (the canonical store, faithful to SPC's `t_summary`/`t_summary_tag`), migrated package-locally in server mode from main — same precedent as `fileids`/`staging`. The protocol layer owns no digest storage (cf. tasks → `taskdb`).

**Phase D2 (UB-side surfacing, 2026-05-26):** `SummaryHandler` now also takes an optional `DigestIndexer` (Config.DigestIndexer; a `*digestindex.Bridge`) and calls it on add/update (index) and delete/group-delete (deindex) so synced digests become searchable. Digests index into the **shared** `note_content`/`note_fts`/`note_embeddings` tables at `digest://<uniqueIdentifier>` (source `"digest"`) — no parallel tables — so they appear in unified search + chat. Nil indexer ⇒ digests still round-trip to the device, just not searchable. Web browse is `internal/web` `/digests` (via `service.DigestService` over `digeststore`).

This package does **not** own storage (tasks → `taskdb`; files/notes → notestore) or human UI (`internal/web`).

## Contracts / invariants

- **Flat envelope.** VOs embed `envelope.BaseVO` so `success`/`errorCode`/`errorMsg` serialize at top level, never under `data`.
- **JWT secret = `Constant.SECRET`** (long ~280-char, `Constant.java:46`), not the 32-char `JWT_SECRET`. Verify is stateless (no Redis, no `exp` enforcement) — real device tokens have no `exp`.
- **`client` mode is regression-safe**: no listener, UB behaves exactly as before. Must stay true.
- **DTO/VO field names verbatim** from decompiled source; cite `<FQN.java>:<line>`. Gotchas: `nextPageTokens` (plural request) vs `nextPageToken` (singular response); `lastModify` (no `d`) in sort DTOs; String-in / Long-out task ids (§8); file EntriesVO uses snake_case wire keys (`path_display`/`content_hash`/`is_downloadable`) with no `@JsonProperty`.
- **File listing reads a dedicated `UB_SPC_FILE_ROOT`, not the OCR `NotesPath`.** The device browses the Supernote storage root (its native `Note/Document/EXPORT/SCREENSHOT/INBOX/MyStyle` layout) directly off the filesystem — `notestore` is not used (it knows only the `Note` subtree and stores SHA-256, while the device wants MD5). Empty `FileRoot` ⇒ listing inert.
- **The device sees only the Supernote source.** The Boox source and its WebDAV tree are invisible to the device — never listed, never merged. This is deliberate ("look exactly like real SPC to the device").
- **`content_hash`/`md5` is MD5**, computed lazily and cached in `spc_file_ids` keyed on (size, mtime). **Folder & file ids are persisted Longs** (the device addresses folders by parent id in `list_folder` and by id in `query_v3`); a null/0 id means the root. **`create_folder_v2` is real** (hardware-validated 2026-05-26): it `MkdirAll`s the requested path under FileRoot, mints the folder's id via the `fileids` registry, and returns `metadata{tag:"folder",id,name,path_display}` — the device requires that server-assigned id to upload notes into the new folder, and an empty-metadata stub made it abort the sync right after create (`query/by/path → create_folder_v2 → synchronous/end`, no upload). Collision: `autorename=false` → `E0322` (matches real server); `autorename=true` → idempotent (returns the existing folder; the real server's rename-with-suffix scheme isn't replicated — the device queries by path first and only sends `autorename=false`). `deleteApi` is still a canned-success stub; listing reads still never mutate the filesystem.
- **Download is a self-issued presigned URL.** `download_v3` returns a `/api/oss/download` URL pointing back at UB; the device fetches it opaquely. UB signs *and* verifies, so the secret (`UB_SPC_OSS_SECRET`, auto-generated first boot) need not match real SPC's hardcoded one — the device never computes a signature. **`GET /api/oss/download` is NOT behind the JWT middleware** (the signature is its only auth); `download_v3`/`generate/download/url` are. The URL encodes the **`path_display`** (root-relative), and the GET handler `SafeResolve`s the decoded path under `FileRoot` (defense-in-depth beyond the signature). A bad/expired signature returns **HTTP 500 + bare plain-text** (`FileDownloadException` → `GlobalExceptionHandler.java:127`), not a JSON envelope.
- **Upload (Phase 4) is a stage-then-verify promote.** `upload/apply` mints an `innerName` (UUID) + a presigned `/api/oss/upload` URL whose `path` encodes the innerName (server-controlled); `POST /api/oss/upload` (tokenless — signature-only, like the download GET) sinks the multipart `file` part to `<FileRoot>/.staging/<innerName>`; `upload/finish` verifies md5+size and atomically renames to the target (`path.Join(finish.path, finish.fileName)` — NOT the apply path), then mints the file id. Bad upload signature → HTTP 500 + plain-text (`FileUploadException`). The OCR kick routes through `pipeline.Enqueue` (UpsertFile → enqueue), **never** `processor.Enqueue` directly (the `jobs.note_path → notes(path)` FK needs the notes row first).
- **`UB_SPC_FILE_ROOT` is a UB-owned tree, NOT the real SPC's data dir.** Decouples UB from any existing SPC install (flip NPM back → real SPC storage pristine). The Supernote source `notes_path` should point inside it (`<FileRoot>/NOTE/Note`) so uploads are OCR-indexed. `.staging`/`.recycle` dot-dirs under FileRoot are excluded from `list_folder`.
- **`.note`/`.pdf` mutations are additive, no legacy teardown** (coexistence): delete = soft-delete to `.recycle/<millis>/<relpath>`; move = atomic rename + `fileids.UpdatePath` (id stays stable); copy = disk copy + fresh id. `move_v3`/`copy_v3` `to_path` is the **full destination path** (§8).
- **The device never sends a folder delete — it deletes a folder's *contents* by file id** (`delete_folder_v3` with the note's id), then mirrors `list_folder` for folder existence (hardware-confirmed 2026-05-26). So a user folder's server-side life is controlled entirely by UB: `delete_folder_v3` and `move_v3` call `pruneEmptyParents` to remove a user folder the mutation just emptied (walks up, `os.Remove`-ing empty dirs, **stopping at native buckets** — rel-depth < 3 under FileRoot: never `NOTE`, `NOTE/Note`, `DOCUMENT/Document`, …). Without this an emptied folder lingers and `list_folder` re-syncs it back — a **zombie folder**. **Known limitation:** an *already-empty* folder the user deletes sends nothing to UB (no file to delete) so UB can't auto-remove it — it re-syncs until removed server-side (web folder-delete is a future surface).
- **Digests (Phase D) are camelCase, soft-delete, idempotent-on-uniqueIdentifier.** The summary DTOs/VOs carry no `@JsonProperty` ⇒ verbatim camelCase Java field names (NOT snake_case like the file DTOs); watch `handwriteMD5` (request) vs `handwriteMd5` (response `SummaryInfoVO`), and `isSummaryGroup`/`isDeleted` as `"Y"`/`"N"` strings (§8). `add/summary[/group]` upserts by `uniqueIdentifier` so a full re-sync never duplicates; `delete/summary/group` cascades a soft-delete to member items. `.mark` blobs live under `<FileRoot>/.digests/` (a dot-dir, excluded from list_folder) and ride the **same** OSS signer + staging area as files — `upload/apply/summary` stages by innerName, add/update promotes (verifying `handwriteMD5`), `download/summary` mints a signed GET. `createTime`/`updateTime` wire form + `partUploadUrl`/chunking are capture-pending (§8).

## Socket.IO gotchas (hard-won on hardware 2026-05-23 — see memory `project_spc_socketio_breakthrough`)

- **Server speaks first.** The device's `io.socket` client connects to the default namespace `/` and does NOT send a Socket.IO CONNECT — it waits to RECEIVE `40` from the server. UB sends `0{...}` (EIO open) **then `40` proactively**. Without it the client never fires `connect` and reconnect-loops forever (looks healthy server-side).
- **Heartbeat is client-driven.** The device sends Engine.IO ping `2`; UB replies pong `3`. UB does **not** send server pings. (Device does NOT send `ratta_ping` — earlier 0b note was wrong.)
- **Event name routes the push, not msgType.** Task sync nudge = emit the **`to-do`** event (device's `TaskService` binds it; `onReceive` → `startTaskSync` unconditionally). `ServerMessage` is the FILE channel; `digest` is digests.
- **`taskListId` must be numeric** (device parses Long); page tokens must be `omitempty` (empty string → device pagination loop).
- **`PUT /schedule/task/list`** body is the `UpdateScheduleTaskListDTO` wrapper (`{taskListId, updateScheduleTaskList:[...]}`), not a bare array.
